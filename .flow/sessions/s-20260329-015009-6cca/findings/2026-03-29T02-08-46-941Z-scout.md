### Reusable Core vs CLI Glue тАФ Detailed Split

#### тЬЕ Would go in `exa-core` package

These have **zero CLI dependencies** тАФ pure data-in, data-out:

```
createClient()          тАФ wraps `new Exa()`, returns Exa instance
filterOptions()         тАФ pure: {opts, keys} тЖТ filtered object
buildContentsOptions()  тАФ pure: opts тЖТ contentsOpts object
```

**What `createClient()` needs:** `EXA_API_KEY` тАФ but the `Exa` constructor reads it automatically from the environment. `createClient()` itself just calls `new Exa()` тАФ no `process.env` in the function body. тЬЕ Safe to keep as-is in core.

Additionally, the **Exa method call shapes** used in each script are implicit core logic:
- The `wantContents` branch logic in `search.ts` and `find-similar.ts` тАФ decides `search` vs `searchAndContents` тАФ is pure business logic that could be wrapped as `search(exa, query, opts)` core functions.
- The `research.ts` `run` subcommand (create тЖТ poll) is a multi-step orchestration that's pure API logic with no reason to be CLI-only.

#### тЭМ Stays in CLI scripts

Everything that touches process/console **by design**:

| Function | Why CLI-only |
|----------|-------------|
| `parseArgs()` | `process.argv` is the contract of a CLI script |
| `requireApiKey()` | `process.env` + `process.exit` тАФ env guard for CLI startup |
| `requireArg()` | `process.exit(1)` on missing arg тАФ fatal for CLI, wrong for library |
| `showHelp()` | Reads script source via `new URL(scriptUrl)`, `process.exit(0)` |
| `handleError()` | `process.exit(1)` тАФ library code should throw, not exit |
| `executeAndPrint()` | `console.log(JSON.stringify(...))` + `process.exit` in `handleError` |

The streaming loop in `answer.ts` (`process.stdout.write`) and the progress message in `research.ts` (`console.error(...)`) are also inherently CLI concerns.

### Findings Summary

- **3 functions are cleanly extractable** into `exa-core` with zero changes: `createClient`, `filterOptions`, `buildContentsOptions`. These have no `process.*` or `console.*` calls.
- **5 functions are CLI-only by construction** (`parseArgs`, `requireApiKey`, `requireArg`, `showHelp`, `handleError`) тАФ they call `process.exit` and must stay in the CLI layer.
- **`executeAndPrint` is a hybrid** тАФ wraps an API call but outputs to `console.log` and delegates errors to `handleError`. In a core package it would become `executeApi<T>(fn) тЖТ Promise<T>` (returns data, throws on error); the print+exit wrapper stays in CLI.
- **`contents.ts` and `research.ts` diverge from the shared `parseArgs()` pattern** тАФ they parse `process.argv` inline. A full extraction would need to normalize this.
- **The `run` subcommand in `research.ts`** (create тЖТ poll in one step) is pure API orchestration with no CLI dependency тАФ it's the strongest candidate for promotion into a core `runResearch()` helper alongside the three already-clean functions.
