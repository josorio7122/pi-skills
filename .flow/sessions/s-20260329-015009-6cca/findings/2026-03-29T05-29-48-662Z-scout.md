### [SKILL WRITER] HIGH ‚Äî Non-standard frontmatter fields
**File:** `SKILL.md`
**Line/Section:** Frontmatter (lines 1‚Äì8)
**Issue:** The frontmatter includes `license` and `metadata` fields (`author`, `version`). The Agent Skills spec defines only `name` and `description` as standard frontmatter fields. Non-standard fields are not parsed by pi and become invisible noise ‚Äî or worse, cause unexpected parser behavior.
**Evidence:**
```yaml
license: Complete terms in LICENSE.txt
metadata:
  author: josorio7122
  version: '1.0'
```
**Fix:** Remove `license`, `metadata.author`, and `metadata.version` from frontmatter. If attribution is needed, add a one-line comment in the body: `<!-- Author: josorio7122 v1.0 -->` or just leave it out entirely.

---

### [SKILL WRITER] MEDIUM ‚Äî No reference files; SKILL.md is both router and payload
**File:** `SKILL.md`
**Line/Section:** Entire file
**Issue:** Progressive disclosure requires SKILL.md to act as a thin router (<150 lines) that gates deeper reference loads. This file currently bundles all guidance inline with no `references/` directory. At 78 lines it stays under the line limit, but the pattern is wrong: if any section grows (more prohibitions, framework-specific guidance, etc.) there is no structure to absorb growth without bloating the router.
**Evidence:** No `references/` directory. All design doctrine, prohibitions, and output rules are inlined.
**Fix:** This is acceptable at current size, but add a `references/` placeholder with at least one file (e.g., `references/aesthetic-system.md`) and gate it: *"Read `references/aesthetic-system.md` when the user requests a specific visual style or asks for style examples."* This future-proofs the skill.

---

### [SKILL WRITER] LOW ‚Äî Description mixes imperative trigger and topic enumeration
**File:** `SKILL.md`
**Line/Section:** `description` field
**Issue:** The spec calls for imperative triggers ("Use when‚Ä¶") with no topic enumeration. The current description enumerates examples parenthetically (`(examples include websites, landing pages, dashboards, React components, HTML/CSS layouts‚Ä¶)`). Enumerating topic examples inflates the description and risks missing unlisted triggers.
**Evidence:**
```
(examples include websites, landing pages, dashboards, React components, HTML/CSS layouts, or when styling/beautifying any web UI)
```
**Fix:** Trim to: *"Create distinctive, production-grade frontend interfaces with high design quality. Use this skill when the user asks to build or style any web UI ‚Äî components, pages, applications, or artifacts. Also use when the user wants to beautify an existing interface or avoid generic AI aesthetics."*

## Lens 3: Prompt Engineering

---

### [PROMPT ENG] HIGH ‚Äî "CRITICAL" and "IMPORTANT" authority signals compete and dilute each other
**File:** `SKILL.md`
**Line/Section:** `## Design Thinking` and `## Prohibitions`
**Issue:** The file uses `**CRITICAL**` and `**IMPORTANT**` as emphasis signals twice. When multiple signals carry the same label, the model treats them as equal weight and neither registers as truly prioritized. This is the "crying wolf" problem in instruction hierarchies.
**Evidence:**
```
**CRITICAL**: Choose a clear conceptual direction and execute it with precision.
‚Ä¶
**IMPORTANT**: Match implementation complexity to the aesthetic vision.
```
**Fix:** Reserve `**CRITICAL**` for one rule only ‚Äî the most likely failure mode. Demote the others to plain bold or restructure them as numbered constraints at the top of the file:
```
**Rule 1 (non-negotiable):** State your aesthetic direction before writing any code.
**Rule 2:** Match implementation complexity to vision ‚Äî maximalism needs elaborate code; minimalism needs restraint.
```

---

### [PROMPT ENG] HIGH ‚Äî No explicit error recovery / fallback instructions
**File:** `SKILL.md`
**Line/Section:** Entire file
**Issue:** There is no guidance for what to do when the model cannot satisfy the aesthetic direction ‚Äî e.g., framework constraint forbids animation, accessibility requirement conflicts with dark brutalist palette, or the user's codebase already has a design system the skill contradicts. Without fallback instructions, the model either silently compromises or ignores constraints.
**Fix:** Add an explicit recovery clause at the end of `## Design Thinking`:
> *"If a technical constraint (accessibility, existing design system, performance budget) conflicts with the chosen aesthetic, prioritize the constraint, name the trade-off explicitly, and adjust the aesthetic accordingly."*

---

### [PROMPT ENG] MEDIUM ‚Äî Degree-of-freedom explosion in aesthetic selection
**File:** `SKILL.md`
**Line/Section:** `## Design Thinking` ‚Üí Tone
**Issue:** Listing 11+ aesthetic directions with no selection heuristic gives the model too many unconstrained degrees of freedom. Without a signal for how to pick, the model defaults to recency bias (whichever direction is listed last or appears most frequently in training). The instruction to "design one that is true to the aesthetic direction" is vague.
**Evidence:**
```
Pick an extreme: brutally minimal, maximalist chaos, retro-futuristic, organic/natural, luxury/refined, playful/toy-like, editorial/magazine, brutalist/raw, art deco/geometric, soft/pastel, industrial/utilitarian, etc.
```
**Fix:** Add a selection heuristic: *"Map the chosen aesthetic to the user's stated purpose and audience. A productivity tool ‚Üí minimal/utilitarian. A creative portfolio ‚Üí editorial/brutalist. A consumer app ‚Üí playful/soft. A fintech dashboard ‚Üí luxury/refined. If no strong signal exists, choose the direction least likely to match AI defaults (i.e., avoid soft/pastel and card-grid minimalism)."*

---

### [PROMPT ENG] MEDIUM ‚Äî Output format underspecified for multi-component output
**File:** `SKILL.md`
**Line/Section:** `## Output`
**Issue:** The output section handles single-file HTML and single React component. It does not specify what to do for multi-page apps, design systems, or cases where scaffolding is "explicitly requested." The phrase "unless scaffolding is explicitly requested" opens a degree of freedom without guidance on what scaffolding output should look like.
**Fix:** Add: *"When scaffolding is requested, produce a file tree first (as a fenced code block), then emit each file in sequence as separate fenced blocks labeled with the file path. Do not produce partial file listings."*

---

### [PROMPT ENG] LOW ‚Äî Inline comment instruction is ambiguous
**File:** `SKILL.md`
**Line/Section:** `## Output`
**Issue:** "Include inline comments only for design decisions that cannot be inferred from the visual output" is subjective. The model cannot reliably determine what a user will infer from visual output ‚Äî this depends on the user's skill level, which is unknown.
**Evidence:**
```
Include inline comments only for design decisions that cannot be inferred from the visual output (e.g., why a specific font pairing was chosen‚Ä¶)
```
**Fix:** Make it objective: *"Add a `/* Design:` comment at the top of the CSS block summarizing the aesthetic direction and any non-obvious choices (font pairing rationale, layout technique). No other inline comments unless the code is non-trivial."*

---

### [PROMPT ENG] LOW ‚Äî Prohibition list has no explicit replacement mandate enforcement
**File:** `SKILL.md`
**Line/Section:** `## Prohibitions`
**Issue:** "Always replace avoided patterns with a deliberate alternative ‚Äî name the choice explicitly" is a good rule but appears *after* the prohibition list and after the directive to interpret creatively. Models tend to anchor on the prohibitions and ignore the replacement instruction placed below them.
**Fix:** Merge the replacement mandate directly into each prohibition:
```
Never use Inter/Roboto/Arial/system-ui as primary display fonts ‚Üí use [name your choice] instead.
Never use purple gradient on white ‚Üí commit to a named alternative palette.
Never use uniform card-grid layouts ‚Üí use [asymmetric grid / editorial split / overlapping panels / etc.] instead.
```

---

### [PROMPT ENG] INFO ‚Äî "State your aesthetic direction in one sentence" is good forcing function
**File:** `SKILL.md`
**Line/Section:** `## Design Thinking`
**Issue:** None. This is a well-placed chain-of-thought anchor that forces commitment before code generation. Worth preserving and possibly strengthening with an example: *e.g., "This interface uses industrial brutalism: exposed grid lines, monospace type, high-contrast black/yellow, zero decorative radius."*

**Blocking issues (fix before shipping):**
1. Strip non-standard frontmatter fields (`license`, `metadata`) ‚Äî they pollute the parser.
2. Collapse `**CRITICAL**`/`**IMPORTANT**` into a single ranked hierarchy ‚Äî competing authority signals degrade instruction fidelity.
3. Add an explicit error recovery clause for when aesthetic intent conflicts with technical constraints.

**High-value improvements:**
- Add an aesthetic selection heuristic to constrain the degree-of-freedom explosion.
- Define scaffolding output format explicitly.
- Add at least one `references/` file and a load gate to make the skill extensible without bloating the router.
